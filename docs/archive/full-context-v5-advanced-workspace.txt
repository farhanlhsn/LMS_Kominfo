CONTEXT UNTUK CODEX - AI-POWERED LEARNING MANAGEMENT SYSTEM LENGKAP
=====================================================================

Versi dokumen: 3.0 - updated dengan gap review, multi-organization SSO strategy, dan plugin-ready architecture
Tujuan dokumen: menjadi konteks utama untuk Codex dalam membangun LMS secara bertahap dari pondasi MVP sampai fitur lengkap skala enterprise.

CATATAN PENTING
---------------
1. Platform harus bersifat general.
2. Jangan gunakan nama instansi, nama daerah, atau nama organisasi tertentu di UI, seed, dokumentasi, variable, maupun copywriting.
3. Gunakan nama produk generic seperti "Learning Platform", "AI Learning Platform", atau "LMS Platform".
4. Platform harus terasa seperti gabungan Coursera dan Moodle, dengan tambahan AI assistant.
5. Phase awal harus menjadi pondasi kokoh agar phase akhir tidak perlu rombak besar.
6. Setiap phase harus menghasilkan aplikasi yang tetap bisa dijalankan.
7. Jangan membuat fitur dummy yang tidak bisa dilanjutkan, kecuali ditandai jelas sebagai placeholder yang sudah punya struktur lanjutan.
8. Semua fitur harus multi-tenant ready, RBAC ready, analytics ready, storage abstraction ready, dan AI ready.
9. Semua API harus versioned dengan prefix /api/v1.
10. Gunakan TypeScript strict mode dan pendekatan modular.

============================================================
A. VISI PRODUK
============================================================

Bangun sebuah AI-powered Learning Management System berbasis web yang menyediakan:
- Course catalog seperti marketplace pembelajaran.
- Course builder seperti LMS modern.
- Materi text, video, PDF, file, link, audio, interactive content.
- Quiz engine lengkap dengan question bank.
- Assignment, rubric grading, project submission, peer review.
- Progress tracking detail.
- Goals, skills, learning path, gamification.
- Sertifikat digital dengan QR verification.
- AI tutor yang menjawab berdasarkan materi course.
- AI tools untuk instructor: generate quiz, summary, flashcard, assignment, rubric, syllabus, outline.
- Discussion forum, announcement, live class, calendar, notification.
- Analytics dashboard untuk learner, instructor, admin, dan super admin.
- Multi-tenant organization management.
- White label branding.
- Payment, marketplace, subscription, coupon, voucher.
- SSO, API key, webhook, audit log, compliance.
- Plugin-ready architecture untuk activity, content, assessment, AI tools, integrations, payment providers, notification channels, video providers, proctoring, analytics, dan certificate requirements.
- PWA, mobile responsive, performance optimized, secure, production-ready.

Produk akhir harus bisa dipakai untuk banyak organisasi, banyak course, banyak instructor, dan banyak learner tanpa perubahan arsitektur besar.

D. PRINSIP DESAIN ARSITEKTUR
============================================================

1. Multi-tenant dari awal
- Resource utama harus punya organizationId jika relevan.
- User bisa punya banyak organization membership.
- Semua query tenant harus difilter organizationId kecuali super admin.
- Gunakan OrganizationContext agar API konsisten.

2. RBAC dari awal
- Jangan hardcode hanya admin/user.
- Gunakan roles, permissions, role_permissions, member_roles.
- Permission harus per module dan action.
- Super admin global.
- Organization admin per organization.

3. Course structure extensible
- Course -> Module -> Lesson -> Activity.
- Activity type harus extensible: TEXT, VIDEO, FILE, LINK, QUIZ, ASSIGNMENT, DISCUSSION, LIVE_SESSION, SURVEY, POLL, SCORM, XAPI, H5P, AI_PRACTICE.

4. Storage abstraction
- Jangan langsung bergantung pada local filesystem.
- Semua upload lewat StorageService.
- File access pakai accessLevel dan signed URL.

5. AI RAG-ready
- Semua materi course bisa diekstrak menjadi AiDocument dan AiChunk.
- AI tutor harus retrieve context dari materi.
- AI generated content harus tersimpan sebagai draft.
- Instructor approve sebelum masuk question bank atau course content.

6. Progress tracking fleksibel
- Simpan progress per activity, lesson, module, course, learning path.
- Completion rule berbasis JSON agar bisa berkembang.
- Video completion, quiz passed, assignment passed harus bisa mempengaruhi progress.

7. Analytics event-based
- Buat LearningEvent sejak awal.
- Record event penting di semua flow.
- Gunakan aggregation table untuk dashboard berat.

8. Certificate verifiable
- Certificate number unique.
- Verification code unique.
- QR code.
- Public verification page.
- Revoke/expire support.

9. API consistent
- Standard response format.
- Standard error format.
- Pagination format.
- Query filters.
- Request validation.

10. Security first
- Password hashing aman.
- Refresh token rotation.
- Rate limit.
- File validation.
- Audit log.
- Secure headers.
- Permission guard.


11. Timezone strategy wajib sejak awal
- Semua datetime disimpan dalam UTC di database.
- UserProfile harus punya timezone default.
- Organization harus punya timezone default.
- Course, Cohort, LiveSession, Assignment due date, Quiz due date, CalendarEvent, dan NotificationSchedule harus menyimpan timezone asal saat dibuat.
- Semua tampilan datetime di frontend dikonversi berdasarkan user timezone, dengan fallback organization timezone.
- API harus menerima timezone eksplisit untuk semua schedule/due date, bukan mengandalkan timezone server.
- Report date range harus jelas memakai organization timezone atau user timezone.

12. Auth provider strategy wajib jelas
- Email/password tetap tersedia.
- Public OAuth/social login harus didukung lebih awal: Google dan Microsoft minimal; GitHub optional.
- Enterprise SSO dipisah dari social login: SAML/OIDC untuk organization-managed login.
- Buat AccountLinking/OAuthAccount entity agar satu user bisa punya beberapa login provider.
- Jangan mencampur konsep public OAuth dengan enterprise SSO.

13. Bot protection dan abuse prevention
- Gunakan CaptchaProvider abstraction untuk register, login berulang gagal, forgot password, dan invite abuse.
- Rate limit berdasarkan IP, email, userId, organizationId, dan API key.
- Simpan security event untuk login failed, captcha failed, password reset request, dan suspicious activity.

14. Realtime strategy wajib dari awal
- Buat RealtimeGateway abstraction.
- Gunakan WebSocket untuk notification realtime, discussion update, direct message, admin event, dan presence jika dibutuhkan.
- Gunakan Server-Sent Events atau streaming response untuk AI tutor agar jawaban AI bisa stream token-by-token.
- Gunakan Redis pub/sub untuk horizontal scaling.
- Fallback polling harus tersedia untuk client yang tidak mendukung realtime.

15. Course delivery mode harus fleksibel
- Platform harus mendukung self-paced course dan cohort/scheduled course.
- Cohort adalah batch/kelas dengan start date, end date, enrollment window, instructor, learner group, due date override, live session khusus, discussion khusus, dan report khusus.
- Enrollment harus bisa linked ke Course saja atau ke Cohort.
- Completion dan certificate rules harus bisa berbeda per cohort.

16. Course publish and approval workflow harus eksplisit
- CourseInstructor role minimal: OWNER, INSTRUCTOR, ASSISTANT, REVIEWER.
- OWNER bisa mengelola penuh course.
- INSTRUCTOR bisa edit content sesuai permission.
- ASSISTANT tidak boleh final publish kecuali diberi permission eksplisit.
- REVIEWER bisa approve/reject jika ditugaskan.
- Organization Admin bisa final approve semua course dalam organization.
- Course workflow minimal: DRAFT, SUBMITTED_FOR_REVIEW, CHANGES_REQUESTED, APPROVED, PUBLISHED, ARCHIVED.

17. Accessibility/WCAG sejak desain UI
- Semua komponen UI harus keyboard navigable.
- Focus state harus jelas.
- Form error harus screen-reader friendly.
- Gunakan semantic HTML dan ARIA label bila perlu.
- Kontras warna harus memenuhi WCAG AA minimal.
- Video harus mendukung caption/subtitle/transcript untuk accessibility.
- Accessibility bukan hanya QA akhir; harus masuk Phase frontend foundation.

18. Video strategy wajib jelas
- Gunakan VideoProviderAdapter agar bisa memakai vendor seperti Mux/Cloudflare Stream atau self-hosted pipeline.
- Untuk production, rekomendasi default adalah vendor managed video agar MVP tidak terbebani encoding kompleks.
- Jika self-hosted, pipeline harus mendukung upload original, queue processing, generate HLS/DASH, thumbnails, duration metadata, subtitle/transcript, signed playback URL, dan cleanup failed jobs.
- Video progress, completion rule, prevent skipping, timestamp notes, dan video analytics harus tidak bergantung pada satu vendor.

19. Payment, multi-currency, dan payout strategy
- Product, Order, Payment, Refund, Invoice, Subscription, Payout harus menyimpan currency.
- Organization bisa punya default currency.
- Marketplace harus mendukung platform fee, instructor revenue share, payout account, payout batch, refund adjustment, dan payout status.
- PaymentProvider dan PayoutProvider harus abstraction-based.
- Exchange rate boleh placeholder dengan interface jelas jika belum implement real conversion.

20. Moderation dan legal compliance wajib ada
- Semua user-generated content seperti discussion, comment, review, project showcase, message, dan profile dapat direport.
- Buat moderation queue, moderation case, moderation action, dan audit log.
- Buat legal pages: Terms of Service, Privacy Policy, Cookie Policy.
- Buat consent UI: cookie banner, terms acceptance, privacy consent.
- Terms/privacy harus versioned; user perlu re-accept jika versi berubah.

21. Bulk operations wajib untuk enterprise LMS
- Admin harus bisa bulk import users via CSV/XLSX.
- Admin harus bisa bulk invite users.
- Admin harus bisa bulk enroll learners ke course/cohort/learning path.
- Admin harus bisa bulk assign role/group/cohort.
- Setiap bulk job harus punya validation preview, failed row report, rollback/undo strategy jika memungkinkan, dan audit log.

22. Proctoring scope decision
- Proctoring tidak wajib untuk MVP.
- Namun exam/quiz architecture harus menyediakan ProctoringProvider abstraction untuk future.
- Implement minimal exam integrity lebih dulu: fullscreen mode, tab switch detection, copy/paste prevention, suspicious event log.
- Webcam/screen monitoring harus optional provider integration dan tidak menjadi dependency core.

23. Provider abstraction rule
- Semua integrasi eksternal harus lewat adapter/interface:
  - StorageProvider
  - VideoProvider
  - LlmProvider
  - EmbeddingProvider
  - PaymentProvider
  - PayoutProvider
  - EmailProvider
  - SmsProvider
  - WhatsAppProvider
  - PushProvider
  - LiveSessionProvider
  - CaptchaProvider
  - SsoProvider
  - ProctoringProvider
- Jangan hardcode vendor tertentu di domain logic.

================================================================================
ARCHITECTURE DECISION: MULTI-ORGANIZATION SSO STRATEGY
================================================================================

This section is mandatory and must be treated as an early architecture decision,
not as a late enterprise-only feature. The LMS is multi-organization, so SSO must
not be implemented as one simple global SSO login. SSO must be organization-aware
from the beginning.

Core principle:
- User is global.
- Organization membership is tenant-scoped.
- SSO provider is organization-scoped.
- Login policy is organization-scoped.
- Role and permission are resolved from the active organization membership.
- A user may belong to multiple organizations.
- A user may authenticate through different identity providers for different organizations.
- Access tokens or sessions must carry active organization context.
- All tenant-scoped queries must be filtered by activeOrganizationId unless the actor is a platform super admin.

Example:
- user@example.com may be an Instructor in Organization A.
- user@example.com may be a Learner in Organization B.
- Organization A may use Microsoft Entra ID SSO.
- Organization B may use Google Workspace SSO.
- The same global User record can have multiple OrganizationMember records and multiple UserIdentity records.

Do not implement enterprise SSO as:
- A single global "Login with SSO" button that does not know the organization.
- A login callback that links users only by email.
- A global role assignment from IdP groups.
- A flow that ignores organization membership status.

Required login modes:
1. Global login with email discovery.
2. Organization-first login via /o/:organizationSlug/login.
3. Subdomain login, for example https://acme.app.example.com/login.
4. Custom domain login, for example https://learning.acme.com/login.
5. Invite-based login.
6. IdP-initiated SSO with signed and expiring RelayState.
7. Public social login, separate from enterprise SSO.

Public social login:
- Google personal/public login.
- Microsoft personal/public login.
- Optional GitHub login.
- Social login authenticates the global user account.
- Social login must not automatically add a user to an organization unless a valid invitation or organization login policy allows it.
- Social login is not the same as enterprise SSO.

Enterprise SSO:
- Organization-scoped.
- Supports SAML and OIDC.
- Each organization may configure one or more SSO providers.
- Each organization may enforce SSO for verified domains.
- Each organization may enable or disable JIT provisioning.
- Each organization may map IdP groups to LMS organization roles.
- Enterprise SSO must never assign platform-level super_admin role.
- Suspended or deactivated organization membership must block access even when SSO authentication succeeds.

Organization resolution strategy:
Before redirecting to enterprise SSO, the system must determine the organization
context from one of the following sources:
- /o/:slug/login route.
- Subdomain.
- Custom domain.
- Email domain discovery.
- Invitation link.
- Signed RelayState for IdP-initiated SSO.

If organization cannot be resolved, do not start enterprise SSO. Ask the user to
select organization or enter email for discovery.

Recommended user experience:

1. Organization-first login:
- User visits /o/acme/login or acme.app.example.com/login.
- The login page shows Acme branding.
- The login page only shows login methods allowed by Acme's OrganizationLoginPolicy.
- If Acme requires SSO, show only the Acme SSO button and hide password login.

2. Email discovery login:
- User visits /login.
- User enters email.
- System checks verified OrganizationDomain records.
- If the email domain maps to one organization with enforced SSO, redirect to that organization's SSO provider.
- If the domain maps to multiple organizations, show organization selection.
- If no organization is matched, show regular password/social login if allowed.

3. Invite-based login:
- User opens invitation link.
- Invitation contains organizationId, email, intended role, allowed login methods, and expiry.
- If the organization requires SSO, redirect to that organization's SSO provider.
- After successful SSO, accept invite and create OrganizationMember if policy allows.

4. IdP-initiated SSO:
- Use signed and expiring RelayState.
- RelayState must include organizationSlug or organizationId, ssoProviderId, returnUrl, nonce, and issuedAt.
- Reject unsigned, expired, or mismatched RelayState.

SSO callback flow:
1. Validate state, nonce, PKCE, issuer, audience, signature, and callback URL.
2. Resolve SsoProvider and Organization from signed state or RelayState.
3. Read stable provider subject from IdP.
4. Find UserIdentity by ssoProviderId + providerSubject.
5. If identity does not exist:
   - If JIT provisioning is enabled, create or link user safely.
   - If invite-only is enabled, require a valid invitation.
   - If domain auto-join is enabled, require verified organization domain.
   - Otherwise reject login.
6. Link or create global User only using safe account linking rules.
7. Create or update OrganizationMember.
8. Apply default role or group-to-role mapping if allowed.
9. Reject access if organization is disabled.
10. Reject access if membership is suspended, deactivated, or removed.
11. Set session.activeOrganizationId.
12. Issue access token for active organization context.
13. Write audit log and SsoLoginAttempt.

Safe account linking rules:
- Do not link an SSO identity to an existing user only because email matches.
- Link automatically only if:
  - IdP email is verified, and
  - organization domain is verified, and
  - organization policy allows auto-linking.
- Otherwise require the user to authenticate with the existing account before linking.
- Never let an unverified email claim an existing user account.

JIT provisioning rules:
- JIT provisioning is organization-scoped.
- JIT provisioning must respect OrganizationLoginPolicy.
- If inviteOnly is true, JIT provisioning requires a valid invitation.
- If autoJoinEnabled is true, the user's email domain must match a VERIFIED OrganizationDomain.
- New JIT users get defaultRoleId or mapped roles.
- If no safe default role exists, use learner or block provisioning.
- JIT provisioning must never create platform super_admin.

Group-to-role mapping rules:
- Group role mapping is optional per organization and per SSO provider.
- IdP groups may map only to organization-scoped roles.
- IdP groups must never map to platform super_admin.
- Organization admin must explicitly configure mappings.
- Record mapping changes in audit logs.
- If group claims are missing, fallback to default role or existing membership roles.

Organization switching:
- A user with multiple active memberships can switch active organization.
- Switching organization must re-check membership status.
- Switching organization must re-resolve roles and permissions.
- Switching organization should issue a new access token or update server-side session context.
- Sidebar, permissions, available features, branding, and data scope must follow activeOrganizationId.
- Cross-tenant data must not remain cached after switching organization.

Access token/session claims:
Access tokens or server-side sessions must include or resolve:
- userId
- activeOrganizationId
- organizationMemberId
- organization role keys or permission version
- sessionId

Do not put large permission lists in long-lived tokens if permissions can change frequently.
Prefer short-lived access tokens and server-side permission resolution or permissionVersion invalidation.

ARCHITECTURE DECISION: PLUGIN-READY LMS STRATEGY
================================================================================

The LMS must be modular and plugin-ready from the foundation phase.

The current architecture is module-based, but a module-based codebase is not automatically plugin-ready. The LMS must include a controlled plugin system so future capabilities such as 3D content, coding exercises, external labs, custom quiz types, proctoring, SCORM/H5P, new payment gateways, new notification channels, and AI tools can be added without rewriting the LMS core.

Core principle:
- The LMS core owns users, organizations, memberships, RBAC, courses, modules, lessons, activities, enrollments, progress, grades, certificates, files, analytics events, audit logs, and tenant isolation.
- Plugins may extend behavior only through controlled extension points.
- Plugins must not bypass RBAC, organization scoping, audit logs, security rules, file access rules, or certificate completion rules.
- Plugins must be organization-aware and permission-aware.
- Plugins must be enableable/disableable per organization.
- Internal plugins are the initial implementation strategy.
- Arbitrary third-party plugin upload, remote JavaScript execution, or dynamic untrusted code loading is not allowed in MVP.

Difference between modular and plugin-ready:
- Modular means the code is organized into modules such as CoursesModule, QuizModule, CertificateModule, and AiModule.
- Plugin-ready means the core has registries, extension points, manifests, permissions, lifecycle events, organization-level enablement, and safe runtime boundaries for optional capabilities.

Initial implementation strategy:
1. Start with internal plugins registered in the codebase.
2. Use PluginRegistry on backend and PluginRendererRegistry on frontend.
3. Store plugin metadata, organization enablement, config, and execution logs in database.
4. Do not allow users to upload arbitrary plugin packages in MVP.
5. Future phases may add plugin marketplace, external plugin review workflow, compatibility checks, signed plugin packages, and sandboxed extension execution.

================================================================================
ADVANCED LEARNING WORKSPACE ARCHITECTURE REQUIREMENTS
================================================================================

The LMS must provide an Advanced Learning Workspace, also called Interactive Learning Mode, inspired by modern platforms such as Coursera. The learning experience must not be a simple static lesson page. It must support flexible multi-panel layouts, side-by-side learning, pop-out windows, dual-monitor-friendly workflows, picture-in-picture video, transcript sync, timestamp notes, AI assistant panels, live coding, interactive preview, 3D preview, and plugin-powered activity panels.

Core goals:
- Make the learning page feel like a modern workspace, not a static content page.
- Support video, article, PDF, quiz, assignment, code exercise, 3D content, SCORM, H5P, AI-assisted learning, interactive simulations, and future plugin activities.
- Allow learners to customize layout based on their learning style.
- Allow learners to study with material on one side and transcript, notes, AI assistant, quiz, code editor, discussion, or preview on the other side.
- Support pop-out panels so learners can move notes, transcript, AI assistant, coding editor, preview, or discussion to another browser window and place it on a second monitor.
- Keep course, lesson, activity, video timestamp, transcript selection, notes, AI context, code submissions, and progress synchronized across panels and pop-out windows.
- Ensure progress tracking remains server-authoritative even when multiple windows are open.
- Ensure assessment security policies can disable AI, notes, transcript, dual-window, or pop-out when required for exams.

Important browser limitation:
- The browser cannot reliably force a window to open on a specific physical monitor.
- Dual-monitor mode must be implemented through pop-out windows and synchronized state.
- The learner can manually drag the pop-out window to their second monitor.

Layout modes:
- standard: curriculum sidebar plus main content.
- side_by_side: main content plus one secondary panel.
- three_panel: curriculum sidebar, main content, and secondary panel.
- focus: only main content, minimal distractions.
- theatre: enlarged video or main activity area.
- split_video_transcript: video and transcript side by side.
- split_content_notes: content and notes side by side.
- split_content_ai: content and AI assistant side by side.
- coding_workspace: instructions, code editor, output, and preview.
- interactive_preview: material plus simulation/3D/preview panel.
- dual_window: one or more panels opened in separate windows.
- popout_panel: individual panel opened in new window.
- picture_in_picture_video: floating video while learner reads, takes notes, codes, or practices.

Layout presets:
- Normal Mode: main material full width with curriculum sidebar.
- Side-by-Side Mode: material plus notes/transcript/AI/quiz.
- Video + AI Mode: video on one side and AI assistant on the other.
- Video + Transcript Mode: video with synced transcript.
- Coding Mode: instructions, code editor, console output, and preview.
- Interactive Preview Mode: content plus 3D/simulation/preview.
- Dual Monitor Mode: supported panels open in new windows.
- Focus Mode: only main learning content without distractions.
- Theatre Mode: large video/content viewing experience.

Split view and side-by-side learning requirements:
- Support two-panel layout.
- Support three-panel layout when screen size allows.
- Panels must be resizable.
- Panels must be collapsible.
- Layout must be responsive.
- Layout must work on desktop, tablet, and mobile.
- User layout preferences must be saved.
- User can restore default layout.
- Each lesson can remember the last used layout state.
- Main content must remain accessible when right panel is closed.
- Curriculum sidebar must be collapsible.
- Secondary panels can include transcript, notes, AI assistant, discussion, resources, quiz practice, flashcards, code editor, console output, live preview, 3D hotspots, or plugin panels.

Dual monitor and pop-out requirements:
- Provide Open in New Window / Pop Out controls for supported panels.
- Supported pop-out panels include:
  - video player
  - transcript
  - notes
  - AI assistant
  - discussion
  - resources
  - flashcards
  - quiz practice
  - coding editor
  - console output
  - live preview
  - 3D preview
  - plugin panels
- Pop-out windows must display the same organization, course, lesson, and activity context.
- Pop-out windows must not bypass RBAC.
- Pop-out windows must not allow cross-tenant data access.
- Pop-out sessions must expire.
- Closing a pop-out window must update workspace state.
- If the main learning tab closes, pop-out windows must gracefully show a reconnect or return-to-learning state.
- Progress must continue to be saved through server APIs.
- Local UI synchronization may use BroadcastChannel API.
- localStorage event fallback should be available if BroadcastChannel is unsupported.
- WebSocket or SSE may be used when server-side real-time state synchronization is required.

Picture-in-picture video requirements:

- Provide mini-player fallback when browser PiP is unavailable.

- Video progress must remain accurate.
- PiP must respect assessment display policy. For exams or restricted assessments, PiP can be disabled.
